# Audit tecnico — librarAIn-server

Data: 2026-06-10 · Analisi read-only · Ogni claim citato come `file:riga` è stato verificato sul codice attuale.

---

## Executive Summary

**Voto complessivo: C+** — Nucleo `src/` funzionale, modulare e testato (27 file di test, ~6k righe), ma con debito operativo serio attorno: secret hardcoded in un file legacy tracciato in git, API HTTP senza autenticazione, ~2.500 file di dati/DB tracciati nel repo, nessuna CI, e un collo di bottiglia di concorrenza critico sul polyindex.

**Top 3 rischi:**
1. Credenziale hardcoded in `RagAIO.py:47` e `:992`, file tracciato in git — leak permanente nella storia.
2. Lock globale su `INDEX.json` tenuto durante chiamate LLM/embedding sincrone (`src/ingestion/polyindex/index_json.py:228-256`) — ingest paralleli serializzati, rischio stallo operativo.
3. Server HTTP senza auth né limite di job concorrenti (`src/api/ingest_http_server.py`) — chiunque raggiunga l'host avvia pipeline GPU/LLM costose.

**Top 3 opportunità:**
1. La suite test estesa rende i refactor sicuri: si può intervenire su orchestrator e polyindex con rete di sicurezza.
2. Eliminare il doppio render PDF (orchestrator + stage1) dimezza il costo I/O dei libri grandi.
3. Le feature mancanti del TODO (metadati in INDEX.json, admin merge POH, TIME_INDEX) si innestano bene sull'architettura esistente.

## Repo Map

**Scopo:** server di ingestion per una biblioteca digitale storica (Fase 1). PDF → OCR (EasyOCR) → raffinamento Vision LLM → Editor LLM → artefatti Markdown per pagina → TOC/INDEX per libro → indici aggregati cross-libro (`data/polyindex/TOC.json`, `INDEX.json`) con matching fuzzy/embedding/LLM dei soggetti (POH). Maturità: **MVP interno avanzato** (PRD dettagliato, no deploy production-grade).

**Stack:** Python 3.x · stdlib `ThreadingHTTPServer` (non FastAPI, scelta MVP documentata) · Pydantic · EasyOCR/PyTorch · OpenAI SDK verso LM Studio locale · SQLite · pypdf/pypdfium2.

**Flusso principale:**

```
POST /api/ingest/submit (multipart)
  → src/api/ingest_http_server.py  (thread per job, SSE progress)
  → src/api/ingest_pipeline_runner.py  (asyncio.run per job)
      ├─ request_validation + gate SQLite (BIBLIOTECA.db)
      ├─ pdf_alignment + page_enumeration
      └─ src/ingestion/orchestrator.py  (_run_pipeline_body, ~337 righe)
           ├─ render PDF → stage1 OCR → stage2 Vision → stage3 Editor
           ├─ output_writer + book/toc/index MD builders
           ├─ toc_index_refine (LLM)
           └─ polyindex: toc_json.py + index_json.py (+ subject_matcher)
```

**Directory chiave:**

| Path | Ruolo |
|------|-------|
| `src/api/` | HTTP server, form parsing, job registry SSE, runner pipeline |
| `src/ingestion/` | Orchestrazione, stage pipeline, builder MD, refine |
| `src/ingestion/polyindex/` | TOC.json / INDEX.json aggregati, subject matcher |
| `src/persistence/` | SQLite: libri (REICAT), pipeline runs, embedding cache |
| `src/core/` | config, log strutturato, client OpenAI con retry/rate-limit |
| `src/models/` | Pydantic `IngestRequest`, `Settings` |
| `tests/` | 27 file, ~5.900 righe |
| `web/index.html` | Unica UI: form di ingest |
| `scripts/` | backfill DB, sort indici, merge PDF |
| `RagAIO.py` | **Monolite legacy (1.108 righe), stack parallelo non integrato** |
| `data/` | output per libro, polyindex, DB — in gran parte tracciato in git |

**Sorprese:** 2.529 file sotto `data/` tracciati in git incluso `data/db/biblioteca.db` (verificato con `git ls-files`); `RagAIO.py` contiene un secondo stack RAG completo con chiave hardcoded; il README descrive una pipeline "solo Stage 1" mentre il codice esegue stage 1–3 + polyindex.

## Improvement Strategy

### Tema 1 — Igiene del repo e dei segreti
**Stato target:** nessuna credenziale nel codice; `data/output/`, `data/db/`, `data/polyindex/` fuori da git; `RagAIO.py` rimosso o archiviato. **Principio:** il repo contiene codice, non dati né segreti. **Done:** `git ls-files | grep ^data/` vuoto (salvo eventuali fixture), zero match per pattern chiave, chiave ruotata se reale.

### Tema 2 — Concorrenza del polyindex
**Stato target:** il matching LLM avviene **fuori** dal lock; il lock copre solo read-merge-write (millisecondi). Limite globale ai job concorrenti. **Principio:** mai I/O di rete dentro una sezione critica. **Done:** due ingest paralleli completano il sync polyindex senza serializzarsi per minuti.

### Tema 3 — Qualità e completezza dei dati INDEX (i fogli)
**Stato target:** `INDEX.json` con `title`/`slug` per libro; perdita parser INDEX.md→json sotto il 3% con righe scartate loggate; tooling admin per merge POH; `TIME_INDEX.json` generato. **Principio:** l'indice è il prodotto — ogni voce persa è valore perso e va resa osservabile. **Done:** i punti del `TODO-ingestion-fix.md` chiusi, delta misurato per libro.

### Tema 4 — Affidabilità operativa minima
**Stato target:** CI che esegue `make test` su push; token statico di auth sull'API; README allineato. **Principio:** calibrato alla maturità MVP — niente OAuth/K8s, solo il minimo che previene incidenti. **Done:** CI verde obbligatoria, endpoint protetti da header token, README corretto.

### Cosa NON sistemare ora (trade-off espliciti)
- **Migrazione a FastAPI** — già pianificata nel PRD (T18.5); il server stdlib regge l'MVP. Rifarlo ora è churn.
- **Tipizzazione Pydantic completa del polyindex (Q2)** — payoff basso finché lo schema è stabile; rimandare a quando si tocca lo schema.
- **WAL SQLite e ottimizzazioni memoria (P5, P6)** — carico attuale (4 libri) non lo giustifica.
- **Lock Windows (Q5)** — deploy solo macOS/Linux.
- **Split dell'orchestrator (A1)** — costoso e rischioso; mitigare aggiungendo le nuove fasi come funzioni separate ben isolate, rimandare lo split vero a dopo la Fase 2.

### Sketch implementativi (top 3)

**M1.2 — Lock fuori dal matching.** Fase 1: con lock, leggere `INDEX.json` e rilasciare. Fase 2: senza lock, eseguire `match_subject` per ogni `raw_subject` contro lo snapshot. Fase 3: ri-acquisire il lock, rileggere il documento, ri-validare le decisioni (un canonical scelto potrebbe non esistere più o esserne nato uno migliore: per le decisioni `new`, ricontrollare con un match fuzzy economico prima di inserire), applicare e scrivere. Gotcha: la ri-validazione deve essere cheap (niente LLM dentro il lock); accettare che in rari casi si crei un duplicato poi mergiabile dall'admin (M2.4).

**M2.2 — Parser tollerante.** In `parse_index_md`: raccogliere le righe non parsate in una lista `skipped` con motivo (nessun separatore, pagine fuori mapping, heading senza pagine); loggarle in WARNING con conteggio e salvarle in `INDEX.skipped.md` accanto a `INDEX.md` per revisione umana; aggiungere euristiche per separatori alternativi (punto e virgola, due punti, spazi multipli prima dei numeri). Gotcha: non allentare al punto da catturare righe di rumore OCR — meglio scartare-e-loggare che inquinare i soggetti.

**M2.4 — Admin merge POH.** Endpoint `GET /api/admin/subjects?min_books=2` che legge `INDEX.json` e ritorna i soggetti multi-libro (+ candidati simili via fuzzy); `POST /api/admin/subjects/merge` con `{target_id, source_ids[]}` che unisce aliases/books/pagine sotto il lock e riscrive atomico. Pagina `web/admin.html` con lista, ricerca e merge. Gotcha: il merge deve registrare un audit (riusare `subject_match_audit` in SQLite) per poterlo ripercorrere.
